# ISO 42010 Meta-Ontology

The standard for architecture descriptions: ISO/IEC/IEEE 42010:2011. It
defines the vocabulary for organising what we already have (per-diagram
Model Kinds) into a coherent description of a system.

This is the **meta** layer. It does not replace the typed Model Kinds
(`architecture`, `er`, `sequence`, …); it composes them.

## Concepts

| Concept | Purpose |
|---|---|
| `Stakeholder` | A party with an interest in (concern about) the system. |
| `Concern` | Something a stakeholder cares about (latency, security, cost). References its stakeholders. |
| `Viewpoint` | A recipe: which concerns it addresses, which Model Kinds it uses, what conventions it follows. |
| `View` | An application of a viewpoint to the system. Contains models. |
| `ArchitectureDescription (AD)` | The top container: all stakeholders, concerns, viewpoints, views, and the models that populate them. |

## Hierarchy

```
ArchitectureDescription
├── Stakeholders                — who needs this AD
├── Concerns                    — what they care about (→ Stakeholders)
├── Viewpoints                  — recipes for views (→ Concerns + Model Kinds)
├── Views                       — instances of viewpoints (→ Viewpoint + Models)
└── Models                      — the actual typed diagrams (architecture / er / sequence / …)
```

Trace links across models (see [trace.md](trace.md)) also live in the AD.

## Schema

Source: `sysatlas/_ontology/iso42010.py`

```python
class Stakeholder(BaseModel):
    name: str
    role: str | None = None
    description: str | None = None

class Concern(BaseModel):
    name: str
    description: str | None = None
    stakeholders: list[str]              # → Stakeholder.name

class Viewpoint(BaseModel):
    name: str
    description: str | None = None
    concerns: list[str]                  # → Concern.name
    model_kinds: list[str]               # e.g. ["architecture", "sequence"]
    conventions: str | None = None

class View(BaseModel):
    name: str
    viewpoint: str                       # → Viewpoint.name
    description: str | None = None
    models: list[str]                    # → AD.models keys

class ArchitectureDescription(BaseModel):
    title: str = ""
    description: str | None = None
    stakeholders: dict[str, Stakeholder]
    concerns: dict[str, Concern]
    viewpoints: dict[str, Viewpoint]
    views: dict[str, View]
    models: dict[str, Any]               # typed by their Model Kind
```

## Validation

Construction enforces all cross-references:

- `Concern.stakeholders` → all must exist in `AD.stakeholders`.
- `Viewpoint.concerns` → all must exist in `AD.concerns`.
- `View.viewpoint` → must exist in `AD.viewpoints`.
- `View.models` → all must exist in `AD.models`.

## How it relates to the existing Model Kinds

The per-type files (`architecture.py`, `er.py`, …) are **Model Kinds**.
An AD's `models` field holds *instances* of those Model Kinds. There is
no forced rename of the existing classes; the meta-ontology simply
wraps them.

The current single-builder usage (`SystemMap` → one `ArchitectureDiagram`)
remains valid. The AD container is for multi-model, multi-view projects
(see [design-principles.md](../design-principles.md) for why splitting
into multiple views is preferred over packing everything in one).

## Example

```python
from sysatlas._ontology.iso42010 import *
from sysatlas._ontology.architecture import ArchitectureDiagram, Component

ad = ArchitectureDescription(
    title="E-commerce platform",
    stakeholders={
        "dev": Stakeholder(name="dev", role="Developer"),
        "ops": Stakeholder(name="ops", role="Platform Operations"),
    },
    concerns={
        "latency": Concern(name="latency", stakeholders=["dev", "ops"]),
    },
    viewpoints={
        "container": Viewpoint(
            name="container",
            concerns=["latency"],
            model_kinds=["architecture"],
            conventions="C4-container with layered grouping",
        ),
    },
    models={
        "storefront": ArchitectureDiagram(...),
        "payments":   ArchitectureDiagram(...),
    },
    views={
        "storefront-view": View(name="storefront-view", viewpoint="container",
                                models=["storefront"]),
        "payments-view":   View(name="payments-view",   viewpoint="container",
                                models=["payments"]),
    },
)
```

Each view fits on its own canvas; the AD knows they describe the same
system.
